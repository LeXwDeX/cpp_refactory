# 变更风险约束 — change-risk

对 C++ 代码执行任何变更（重构或新功能）之前，必须评估风险并确认缓解措施就绪。

---

## 硬约束

1. **未评估不变更** — 对目标代码区域执行任何修改工具之前，必须先完成风险评估。违反即停止，回到评估。
2. **高风险维度必须有缓解措施** — 任何风险维度达到 HIGH 时，必须在变更前列出具体缓解措施。无措施则不变更。
3. **必需工具未就绪 → 不变更** — ASan/TSan/gtest/gcovr 等被评估为必需但未安装时，记录工具缺口（GAP），不降级继续。
4. **影响范围必须可追溯** — 变更涉及的每个符号必须有 `codegraph_impact` 或等效分析记录。无记录视为影响未知，按 HIGH 处理。

---

## 风险维度

| 维度 | HIGH 判定条件 | 必需缓解 |
|------|--------------|----------|
| **耦合度** | 共享全局变量 ≥3 或 extern 引用 ≥5 | 特征化测试锁定现有行为 |
| **宏复杂度** | 条件编译嵌套 ≥3 层或功能性宏 ≥5 个 | 宏展开结果人工确认 |
| **内存管理** | 裸 new/delete ≥3 处或 void* 使用 | ASan 必须可用 |
| **并发** | 多线程共享状态或锁粒度不明 | TSan 必须可用 |
| **测试覆盖** | 覆盖率 <50% 或无特征化测试 | 先写特征化测试再变更 |

---

## 输出契约

风险评估完成后产出风险卡片，必须包含：

```
变更目标：{文件}:{行范围} — {符号名}
变更类型：{重构 | 新功能 | Bug 修复}
C++ 标准：{CPP_STANDARD}

风险维度：
  耦合度：{LOW|MED|HIGH} — {判定依据}
  宏复杂度：{LOW|MED|HIGH} — {判定依据}
  内存管理：{LOW|MED|HIGH} — {判定依据}
  并发：{LOW|MED|HIGH} — {判定依据}
  测试覆盖：{LOW|MED|HIGH} — {判定依据}

综合风险：{LOW|MED|HIGH}
必需工具：{列表} — 就绪状态：{全部就绪 | 缺口列表}
缓解措施：{每个 HIGH 维度的具体措施}
```

综合风险 = 任一维度 HIGH → HIGH；无 HIGH 但 ≥2 个 MED → MED；其余 LOW。

---

## 记忆写入

```
subject = "{变更目标符号}"
predicate = "risk"
object = "{综合风险}|{变更类型}|{评估日期}"
```

HIGH 风险变更追加一条到持久记忆抽屉：
- wing = "risk-register"
- room = "{项目名}"
- content = 风险卡片摘要 + 缓解措施

---

## 纠错规则

- **评估过程中发现未知依赖** → 停止评估，回到 code-analysis 补充分析，不凭假设继续。
- **必需工具不可用** → 记录 GAP，停止变更，不跳过该维度评估。
- **同一区域评估 3 次仍无法确定风险** → 停止，上报用户，标注"需人工审查"。

---

## 适用场景

| 场景 | 评估重点 |
|------|----------|
| 重构：分区规划 | 每个分区变更前评估，结果写入 PARTITION_LEDGER |
| 开发：新功能设计 | 评估插入点对现有代码的影响 |
| 开发：API 设计 | 评估公共 API 变更的下游影响 |
| Bug 修复 | 评估修复是否引入新风险 |
