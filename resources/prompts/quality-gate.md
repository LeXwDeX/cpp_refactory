# 质量门禁 — quality-gate

C++ 代码变更（重构或开发）完成后，必须通过质量门禁才能标记为 VERIFIED。任何一项不通过则不交付。

---

## 硬约束

1. **编译零新增错误** — 编译输出中不得出现变更前不存在的错误。新增警告视为错误处理。
2. **测试零回归** — 所有现有测试必须通过。特征化测试（如有）必须通过，证明行为未改变。
3. **C++ 标准不越界** — 不得引入 `CPP_STANDARD` 不支持的语言特性。
4. **所有权规则** — 禁止引入 `std::shared_ptr`（除非原代码本身就是引用计数的）。优先 `std::unique_ptr` 或栈对象。C++03 项目用 RAII wrapper。
5. **性能零退化** — 禁止增加不必要的虚函数调用（V-Table 消耗）或不必要的堆分配。热点路径的变更必须保持零额外开销。
6. **宏保留** — 所有 `#ifdef` 条件编译路径必须保留。可以将宏抽象为策略模式，但必须保留编译期条件分支能力。
7. **cast 标注** — 所有 `const_cast` / `reinterpret_cast` 必须附带 `// why:` 注释说明理由。
8. **驾车原则** — 无超出任务要求的代码改动。每行变更可追溯到变更计划。未引入未被要求的抽象层。

---

## 安全检查（不可跳过）

| 检查项 | 判定标准 |
|--------|----------|
| 迭代器失效 | 无新增风险 |
| 悬空引用 | 无新增风险 |
| use-after-free | 无新增风险 |
| 动态分配所有权 | 所有 new/malloc 有明确所有者 |
| 接口变更 | 修改了接口 → 所有调用方已同步更新 |
| 相邻区域 | 相邻代码的接缝未被无意修改 |

---

## 静态分析验证

变更涉及的文件必须通过：

```
clang-tidy -p . {文件}
cppcheck --enable=all --suppress=missingIncludeSystem {文件}
```

新增告警视为不通过。

---

## 输出契约

质量门禁通过后产出验证报告：

```
变更目标：{文件}:{行范围} — {符号名}
变更类型：{重构 | 新功能 | Bug 修复}

编译：PASS（零新增错误/警告）
测试：PASS（{N} 个测试通过，0 失败）
静态分析：PASS（clang-tidy 0 新增，cppcheck 0 新增）
安全检查：PASS（6/6 项通过）
C++ 标准合规：PASS（未引入 {CPP_STANDARD} 以外特性）
驾车原则：PASS（{N} 行变更，全部可追溯）

判定：VERIFIED
```

任一 FAIL → 列出具体问题，修复后重新验证。

## 记忆写入

```
subject = "{变更目标符号}"
predicate = "verified"
object = "{变更类型}|{验证日期}"
valid_from = "{日期}"
```

追加一条到持久记忆抽屉：
- wing = "patterns"
- room = "cpp-{重构|开发}"
- content = 本次变更的可复用经验（一段话：做了什么、踩了什么坑、下次注意什么）

---

## 纠错规则

- **编译失败** → 修复编译错误，不修改测试来适配错误代码。
- **测试失败** → 区分：代码 bug（修代码）vs 测试本身过时（修测试，但需用户确认）。
- **静态分析新增告警** → 修复告警，不 suppress（除非确认为误报并标注理由）。
- **同一变更 3 次验证仍 FAIL** → 停止，质疑变更方案本身，上报用户。

---

## 适用场景

| 场景 | 额外检查 |
|------|----------|
| 重构：分区完成 | 确认 PARTITION_LEDGER 状态更新为 VERIFIED |
| 开发：新功能完成 | 确认新功能有对应测试覆盖 |
| 开发：API 变更 | 确认所有下游调用方已适配 |
| Bug 修复 | 确认修复有回归测试锁定 |
